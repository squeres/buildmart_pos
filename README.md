# StroyKassa

StroyKassa — монолитное PHP/MySQL-приложение для строительного магазина с кассой, товарами, многоскладским учетом, поступлениями, приемкой, перемещениями, инвентаризацией, продажами и профилями пользователей.

Проект развивается как SSR-приложение без тяжелого frontend-фреймворка: основная логика находится в PHP-модулях, а интерфейс дополняется точечными JS-сценариями.

Важно: в кодовой базе все еще встречаются legacy-имена `buildmart`, домен `buildmart.local` и имя БД `buildmart_pos`. Это техническое наследие и не мешает пользовательскому бренду StroyKassa.

---

## Что умеет система

- Касса (POS) с обычным чеком, скидками по строке и по чеку, несколькими способами оплаты и контролем смен.
- Продажи, история продаж, просмотр продажи, печать чеков и связанные документы.
- Каталог товаров с несколькими единицами измерения и несколькими типами цен.
- Многоскладской учет через `stock_balances`.
- Поступления, приемка, перемещения, списания и корректировки.
- Полноценная инвентаризация с поиском по штрихкоду, SKU, названию и алиасам.
- Создание нового товара прямо из инвентаризации.
- Минусовые остатки в кассе и складском учете продаж.
- Клиенты, поставщики, категории, склады и бизнес-сущности.
- Профиль пользователя: имя, email, телефон, язык, пароль и PIN.
- Авторизация по паролю и PIN.
- Роли и персональные переопределения прав доступа.
- Русский и английский интерфейс, русский язык по умолчанию.

## Ключевые особенности

### Многоскладской учет

- Остатки по складам хранятся в `stock_balances`.
- История движений хранится в `inventory_movements`.
- Продажи, приемка, поступления, перемещения и инвентаризация работают в привязке к складу.

### Единицы измерения

- Базовая единица товара хранится в `products.unit`.
- Дополнительные единицы хранятся в `product_units`.
- В интерфейсе пользователь может работать в выбранной единице, а учет сохраняется в базовой.
- Для цен по единицам используются `product_unit_prices`.

### Инвентаризация

- Отдельный модуль инвентаризации находится в `modules/inventory/count.php`.
- Поддерживается быстрый поиск товара по штрихкоду, SKU, названию и алиасам.
- Можно создать товар прямо из сценария инвентаризации.
- Для каждой позиции в правой панели можно выбрать единицу измерения фактического количества.
- При сохранении фактическое количество конвертируется в базовую единицу и устанавливает остаток на складе.
- Движения инвентаризации пишутся в `inventory_movements` с типом `inventory`.

### Минусовые остатки

- Продажа при нулевом или недостаточном остатке поддерживается.
- Если товар физически есть, но поступление еще не внесено, продажа может увести остаток в минус.
- В POS товары с `0` и отрицательным остатком остаются доступными в поиске и для продажи.
- Поведение контролируется настройкой `allow_negative_stock` и по умолчанию включено.

### Права доступа

- Базовые права задаются ролью.
- Для конкретного пользователя можно настроить персональные override-права поверх роли.
- Это позволяет, например, оставить кассиру просмотр клиентов, но запретить создание или удаление.

### Локализация и профиль

- Русский язык используется по умолчанию.
- На логине можно выбрать язык, и он применяется ко всему сайту после входа.
- Сохраненный язык профиля становится основным языком пользователя.
- В профиле пользователь может сменить язык, пароль и PIN.

## Быстрый старт

### 1. Установка зависимостей

```bash
composer install
```

### 2. Настройка окружения

Скопируйте `.env.example` в `.env` и заполните значения:

```env
DB_HOST=localhost
DB_NAME=buildmart_pos
DB_USER=root
DB_PASS=change_me
BASE_URL=https://buildmart.local
```

Конфигурация подтягивается в [config/config.php](C:/OSPanel/home/buildmart/config/config.php).

### 3. Создание базы

Для новой установки обычно достаточно импортировать:

```bash
mysql -u root -p buildmart_pos < database.sql
```

### 4. Веб-сервер

Для локальной разработки в OSPanel/OpenServer обычно используется:

- URL: `https://buildmart.local`
- Корень проекта: `C:\OSPanel\home\buildmart`

В проекте уже есть `.htaccess`, поэтому для Apache дополнительная базовая маршрутизация обычно не нужна.

## Первая настройка

После чистого импорта БД проверьте:

- наличие хотя бы одного пользователя с ролью администратора
- настройки магазина в разделе `Настройки`
- склады, категории, типы цен и базовые справочники

Если в базе еще нет администратора, его нужно создать вручную.

Пример генерации bcrypt-хеша:

```bash
php -r "echo password_hash('StrongPassword123!', PASSWORD_BCRYPT), PHP_EOL;"
```

## Авторизация и пользователи

### Вход

- Вход по email и паролю.
- Быстрый вход по PIN-коду.
- Язык можно выбрать уже на логин-странице.

Пользователь может изменить только свои данные:

- имя
- email
- телефон
- язык интерфейса
- пароль
- PIN

Роль, системные права и чужие данные через профиль не меняются.

### Роли и права

Сейчас используются:

- роли
- персональные override-права пользователя

Это позволяет гибко настраивать доступ к отдельным разделам и действиям без ручного редактирования БД.

## Безопасность

В проекте уже используются:

- prepared statements через PDO
- CSRF-защита
- `password_hash()` для паролей
- PIN в защищенном хранении
- проверка прав на уровне серверных модулей
- экранирование вывода через helper-функции

Для production дополнительно рекомендуется:

- хранить реальные доступы к БД только в `.env`
- использовать HTTPS
- ограничить доступ к служебным директориям
- регулярно делать бэкапы базы

## Важные замечания для разработки

- Остатки и движения считаются в базовой единице товара.
- Пользовательские единицы — это слой ввода и отображения.
- Нельзя подменять серверные проверки одними UI-ограничениями.
- Для новых пользовательских строк используйте локализацию, а не хардкод.
- Если обновляете старую БД проекта, сначала проверьте применены ли миграции `016`–`019`.
